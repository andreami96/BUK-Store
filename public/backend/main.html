Specification

Purpose


The BUK Store application was developed during the Hypermedia Application (Web and Multimedia) course of the Politecnico di Milano. The main purpose of this application is to develop a web application for an online book store and its multimedia content about books and authors. In particular the web application should handle 3 main entities: Books, Authors and Events. Each book can be written by different authors, and of course each author has probably written more than one book. There are also events during which a book is presented.
So the user should be able to surf the BUK Store application, finding the useful information about each book, event or author, but also he or she should be able to reserve some books and to see his/hers purchases.


Web Architecture


The BUK Store Web Application is divided into the common Client-Server pattern. The client is represented by the user's web browser, and for that the frontend part of the application is written with HTML, CSS and JavaScript, in this way the most common browsers can handle and render properly the frontend of this application.

Focusing on the Server, it is developed using the famous Node.js framework and it is deployed on a Heroku machine.

The Server is divided into 4 main component, which are in charge of fulfill a specific function:


As you can notice from the above diagram, the four part are the model, the controller, the route handler and a static part, used to serve the HTML, CSS or JavaScript files.

The Model

This is the core part of the data of BUK Store application. This module is in charge of creating the entire structure of the Database, both each one of the tables (which they will be described later in this document) and also the relationship between them. In the Model package of the application you can find various folders, each one contains the files which tells how the tables should be stored inside the PostgreSQL database.
Then you can find a special file called sequelize.js, which is the main entry point of all the Model. Each table is actually created in this file, and each relation is mapped here, thanks to the Sequelize instructions table.belongsTo or table.belongsToMany. This code is actually executed before the Server starts up, in this way it is assured that the Database is created before the application starts.

The Controller

The database is nothing without the controller, and the purpose of this module is exactly to fulfill the REST API specification, querying the Database to obtain results and to pack them into nice objects used to be sent to the client.
This module is divided into various categories, one for each request the Server can handle. In this files you will find 2 main types code, one used to actually query the Database to retrieve the information requested, and another one used to pack this data into JSON objects, which will be sent to the client by the Route Handler.
The controller is also responsible to check the parameters of the client's requests, in order to avoid some unpleasant behaviours, and is also responsible to check if the user is logged in or not (done inside the checkAuthMiddleware function), in order to properly handle the user's session, the user's cart and the user's books reservation

The Route Handler

This component is made thanks to the Express Framework, part of the Node.js suite. This part of the web application is the entry point for all the requests which come to the server.
The route handler component is implemented inside the Route package, which contains the entire set of function used to handle the endpoints of the API. All this function are the common ones used in the express environment:
router.get('/', function(req, res, next) { ...code... });
The Route Handler component, for the sake of maintainability, is splitted into 2 main javascript file, the staticRouter.js and the apiRouter.js, which are used to handle, respectively, the static pages and the API endpoints. More specifically, the Static Router is used to handle different URLs, and it only sends to the client the requested page, without any kind of rendering; while the API Router is used to fulfill the endpoints requests, giving back to the client the JSON object with the data requested.
To be more sure that no pages can be rendered from the server, actually there isn't any kind of component installed in our application which can be render a page (I.E. the EJS view engine), so the Server is only able either to retrieve static pages (HTML, CSS and JavaScript client side) or to construct JSON objects to be sent to the client through the API endpoints


API


The whole architecture fulfill the REST API specification, indeed, as you can see from the SwaggerUI page, the only accepted request are GET, POST, PUT and DELETE. All the request made to the server will be responded by a JSON object (or an array of JSON objects), which contains either the data requested or an error message, which specifies the HTTP error code and a description of the occurred error. Each response gives one of the following HTTP code:
  • 200: The request is successful, and the Server return a proper object
  • 201: The request is successful, and a new User is created inside the DB. You should received a mail by the Server with a activation link for that account
  • 400: The request isn't well written. Probably the parameters you have given either in the query part or in the body part aren't properly written, or maybe there is a required parameter which is actually missing
  • 401: This endpoint belongs to the protected one, so it is required the login before accessing this endpoint. If you receive this error, it means that either the login isn't successful or you are trying to access to a private endpoint without a valid session token
  • 404: The path specified in the URL doesn't exists
  • 409: There is a conflict inside the Database. This happens when you are trying to register a user which has the same email address of another user already registered in the application
  • 500: The Server has encountered an error during the processing of your request
To adhere more to the REST API specification, the HTTP verbs used in the BUK Store application have the usual "REST" meanings:
  • GET: Request used to retrieve information from the Database, it is a Safe call (so it doesn't change the Data status) and it preserves the Idempotency property (so applied two times consequently retrieves the same result)
  • POST: Request used to create a new Datum inside the Database. In the BUK web app is used to create new user, to add a new product to the user's cart or to create a reservation for the current user. It isn't a safe call, since it changes the status of the DB, and of course it isn't idempontent, because two consequently call changes two times the status of the Database
  • PUT: Request used to change and update a Datum inside the Database. In our application, is used only to change a book inside the user's cart, in order to be able to change the quantity of desired books (this request isn't able to delete a book from the cart). The PUT request isn't a Safe call, since its purpose is to update a resource, but it is idempotent, since applied two times, of course with the same parameters, it update the same resource with the same values
  • DELETE: Request used to delete and a resource from the Database. In the app is used only when a user wants to delete a book inside his/hers cart. Of course this isn't a Safe call, as you can imagine, but it preserves the Idempotency property, since apply two times the DELETE to the same object doesn't delete the object twice, obviously
To have a more faithful implementation of the REST APIs, we have decided also to keep, as much as possible, the same name convention used in the REST specification, so a great number of endpoints follows the usual convention, such as /api/v1/books which retrieves all the books inside the database, or /api/v1/authors/1/books, which is used to retrieve the books written by the author with identifier 1. But, sometimes, some other endpoints are introduced, to be more specific, for instance there is the /api/v1/me endpoint used to retrieve the information about the the user which is actually logged in, or the api/v1/me/cart, which doesn't follow the usual name convention (in that case it should be carts, plural, not singular), but this decision was taken since each user as only one cart, so the singular emphasizes this concept.
Finally, the whole API system is designed in order to make the Server able to handle all the possible requests that a user can do, with the minimum amount of endpoints and the minimum amount of bandwidth wasting, so, for instance, when the books are requested, since the book object is quite large, and inside the database there can be hundreds of them, the decision made was to create the JSON object with the minimum amount of information (so, in this case, only the title and the ISBN), and to shift the responsibility of composing a bigger object to the more soecific endpoint /api/v1/book/{ISBN}, which is called only when all the book's information are required.

OpenAPI Data Model

The model objects used inside the API are the following:
  • MainGenre: Describes the Top Genre inside the Database (such as Novel or Poetry)
  • Genre: Describes the Sub Genre inside the Database (such as thriller novel)
  • Theme: Describes the themes inside the Database (such as love or friendship)
  • Author: Object used to describe the properties of an Author
  • Book: Object used to describe a book
  • Event: Object used to handle an event information
  • Review: Model which contains a review done by the critics
  • User: Model of an User inside the application
  • Cart: Object which handles the books inside a user's cart
  • Reservation: Model of a purchase of the user
  • GenericPage: TOBEDONE

Database Model



The above diagram is an Entity Relationship (for the sake of brevity ER) Diagram, used to describe more deeply the Database of the BUK application. Diving into this diagram, they can be found various main entities (with a rectangular shape), such as the Book, the Event, the Author or the User, and various relations (with a diamond shape) between this entities, like the Cart, the Reservation or the WrittenBy.Depending on the actual type of relation (if it is 1:1 or 1:N or N:N), some of them will be deployed as a real table, while others will be implemented simply adding a reference inside other tables, already present. To give you an idea, for instance the Cart relation, since it is a relation between Book and User of type N:N, is implemented as a real table inside the Database, while the relation Has between Review and Book, which maps N different reviews with only 1 book, will be implemented adding a reference to the ISBN of the book inside the Review table.
